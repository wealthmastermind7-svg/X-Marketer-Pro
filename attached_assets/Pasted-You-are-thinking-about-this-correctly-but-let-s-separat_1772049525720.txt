1️⃣ The Core Truth About Replit Deployments

In Replit Deployments:
	•	The platform injects process.env.PORT
	•	Your app must listen to that exact port
	•	You should not hardcode any port
	•	You should not manually fight port mappings unless absolutely necessary

If .replit defines custom port mappings (like 8081 → 80), then:
	•	Replit expects your server to listen on 8081
	•	Health checks will target the internal port mapped to external 80

⸻

2️⃣ The Real Root Cause

From everything you described:
	•	.replit sets PORT=5000
	•	Port mapping includes:
	•	8081 → external 80
	•	5000 → external 5000
	•	Health checks hit external 80
	•	That routes to internal 8081
	•	But server listens on 5000

That mismatch causes the failure.

⸻

3️⃣ Your Two Clean Options

You must pick one. Not both.

⸻

✅ Option A (Professional & Clean — Recommended)

Remove custom port mappings entirely.

Let Replit auto-detect the port.

Requirements:

Server:

const PORT = process.env.PORT || 5000;
app.listen(PORT);

Run command:

npm run server:prod

No PORT= override.

No custom port mapping.

No .replit env PORT override.

This is the clean production approach.

⸻

⚙️ Option B (If You Must Keep 8081 → 80 Mapping)

Then the server must listen on 8081.

That means:

Run command:

PORT=8081 npm run server:prod

And ensure:

const PORT = process.env.PORT;

No fallback to 5000 in deployment.
